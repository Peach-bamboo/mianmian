# 5 分钟面试演示脚本

## 第 1 分钟：项目与问题

> 这是我用 AI 协同开发完成的前端学习小程序。它不是单纯的页面生成项目，我重点解决了题库云化、用户状态同步、Markdown 内容能力和微信小程序自动验收。

展示：

- 小程序首页
- 题库总览
- 题目详情
- 可视化实验室

## 第 2 分钟：架构设计

打开：

- `docs/portfolio/architecture-case-study.md`
- `src/services/questionBank.js`
- `src/domain/questionBankCore.cjs`

讲解：

> 我先把题目正文、用户状态和行为记录分开。页面只依赖题库服务层，服务层负责云端、缓存和本地兜底。因此云函数未部署或数据库为空时，页面仍能工作。

## 第 3 分钟：Bug 根因分析

打开：

- `docs/portfolio/root-cause-analysis.md`
- `tests/question-bank-core.test.cjs`

讲解：

> 自动化发现详情页能打开，但展开解析后没有 Markdown。最终根因不是组件，而是旧缓存覆盖新版正文。我修改了数据合并模型，并增加测试保证旧缓存只能继承用户状态。

## 第 4 分钟：AI 辅助测试

执行：

```bash
pnpm test
```

讲解：

> 测试不是让 AI 对着实现生成。我先定义数字/字符串 ID 兼容、状态保留、旧缓存不能覆盖正文等行为契约，再让 AI 补充用例，最后检查测试能否在错误实现上失败。

## 第 5 分钟：完整工程验证

执行：

```bash
pnpm portfolio:verify
```

补充展示：

- `scripts/mp-page-check.cjs`
- `scripts/mp-probe.cjs`
- `scripts/validate-question-bank.cjs`

总结：

> 我使用 AI 的价值不是更快地产生代码，而是更快地枚举方案、验证假设和形成工程资产。最终结果仍然通过测试、构建、页面断言和云端数据证据来负责。

## 面试官追问时

### AI 写错了怎么办？

> 我把 AI 的输出当作待验证假设，不作为完成依据。构建通过只能证明语法正确，页面断言和数据探针才能证明行为正确。

### 为什么不用完整 Markdown 库？

> 小程序环境和包体积有限，当前只需要高频语法，因此先采用轻量结构化解析。未来语法需求扩大时再评估成熟库。

### 这套架构有什么不足？

> 当前仍有包体积警告，云函数自动化测试也不够完整。下一阶段会做分包、云函数契约测试和 CI 集成。
